Einfache Textseiten
Das Herzstück von HTML bleiben nach wie vor die Anweisungen zur Strukturierung von Textdokumenten. Überschriften, Absätze, Abbildungen, Tabellen, Verweise und so weiter. HTML kennt für fast alles einen passenden Befehl.
Komplexe Webseiten
Moderne Webseiten sehen natürlich längst nicht mehr so einfach wie früher aus. Oftmals steht auch gar nicht ein langer Fließtext im Vordergrund, sondern die visuell ansprechende Aufbereitung der Informationen und multimedialer Inhalte.
Browser Games
Seit es Computer gibt, gibt es auch Computerspiele. Wirklich leistungshungrige Spiele laufen natürlich weiterhin direkt auf der Hardware. Da aber inzwischen selbst 3D-Grafik im Browser möglich ist, lassen sich viele Spiele inzwischen online spielen. Siehe zum Beispiel The WebGL Bookcase für ein einfaches aber beeindruckendes Beispiel.
Browser Apps
Dank HTML5 und modernen JavaScript-APIs handelt es sich beim Browser längst nicht mehr um ein einfaches HTML-Anzeigeprogramm sondern um eine vollständige Laufzeitumgebung für grafische Anwendungen. Immer mehr Anwendungen werden daher in Form von SaaS-Cloudangeboten direkt im Browser genutzt. Dropbox, Google Documents oder Microsoft Live sind nur ein paar der bekanntesten Beispiele, aber auch On-Premise-Anwendungen, die nicht in der Cloud laufen, haben oft eine webbasierte Benutzeroberfläche.
Smart TV Apps
Dieses Beispiel mag auf den ersten Blick etwas überraschen. Denn tatsächlich galten Fernseher die längste Zeit als nicht-programmierbare Geräte, die einfach nur das tun, was sie eben tun (vgl. Eingebettetes System). Neuere Geräte erlauben aber durchaus die Installation eigener Apps und inzwischen hat sich HTML hier als UI-Technologie gegenüber anderen Programmiersprachen durchgesetzt, da dies einen besonders leichten Ein- und Umstieg ermöglicht.
Mobile Apps
Bei mobilen Apps für Smartwatch, Smartphone und Tablet gibt es im Prinzip zwei Arten: Die Anwendungen, die mit dem SDK des jeweiligen Herstellers entwickelt wurden und daher nur auf einer bestimmten Geräteklasse laufen (z.B. Android oder iOS) und die, die mit HTML realisiert wurden und daher auf vielen verschiedenen Geräten lauffähig sind. Schätzungsweise handelt es sich dabei um rund die Hälfte aller mobilen Anwendungen. Werkzeuge wie Apache Cordova oder sein kommerzieller Ableger Phonegap erlauben es, komplette Apps nur mit HTML, CSS und JavaScript zu entwickeln.

Du siehst, ohne zu übertreiben kann man sagen, dass HTML inzwischen zu einer universellen UI-Technologie geworden ist. Egal ob groß oder klein, HTML (in Verbindung von CSS und JavaScript) ist auf jedem Gerät daheim und eignet sich zur Gestaltung fast jeder beliebigen Benutzeroberfläche.

Sprachen und Formate rund um HTML

HTML (HyperText Markup Language)

  • Auszeichnungssprache zur Definition der darzustellenden Inhalte
  • Beschreibt nur den Inhalt und seine Bedeutung, nicht aber das Aussehen
  • Überschriften, Absätze, Aufzählungen, Tabellen, Bilder, …

CSS (Cascading StyleSheets)

  • Sprache zur Definition der Layout- und Darstellungsoptionen einer HTML-Seite
  • Bezieht sich auf die Inhalte eines HTML-Dokuments und legt ihre Darstellung fest
  • Farben, Hintergründe, Umrandungen, Positionierung, Transparenz, Schatten, …

SVG (Scalable Vector Graphics)

  • XML-basiertes Grafikformat für skalierbare Vektorgrafiken
  • Kann mit CSS und JavaScript umfangreich zur Laufzeit angepasst werden
  • Ist daher gut zur Definition interaktiver, grafischer Elemente geeignet
  • Rechtecke, Kreise, Linien, Kurven, Flächen, Füllungen, …

JavaScript

  • Programmiersprache, die innerhalb des Browsers ausgeführt wird
  • Ermöglicht es, auf Ereignisse wie Mausklicks zu reagieren
  • Kann die dargestellten Inhalte einer Seite zur Laufzeit ändern
  • Bietet ein umfangreiches API zur App-Entwicklung

Dass die Beispiele auf der ersten Folie nur auf HTML basieren, ist nicht ganz richtig. Denn streng genommen handelt es sich bei HTML nur um eine Sprache zur Strukturierung der Inhalte. Zwar werden der grundlegende Aufbau einer Seite sowie alle darin enthaltenen Informationen in HTML definiert, ohne ein paar Freunde geht es dann aber doch nicht. 😎

Jede HTML-Webanwendung besteht daher aus einer Kombination der folgenden Programmiersprachen, die Hand in Hand zusammenspielen:

HTML (HyperText Markup Language)
Mit HTML werden die darzustellenden Inhalte definiert und die „Bereiche” einer Seite festgelegt. Hierfür werden in einer HTML-Datei alle Textinhalte wie Überschriften, Absätze oder Abbildungen festgelegt und mit Namen versehen, um sie später zum Beispiel als Kopfbereich oder Side Bar darstellen zu können.
CSS (Cascading StyleSheets)

Das Aussehen der HTML-Inhalte wird in einem sogenannten Stylesheet definiert. Dadurch werden Inhalt und Aussehen getrennt, so dass ein einemal erstelltes Layout für mehrere HTML-Dateien verwendet oder leicht ausgetauscht werden kann.¹ Hier kommen dann die Namen ins Spiel, die den einzelnen Elementen in den HTML-Dateien gegen wurden, um ihr Aussehen zu definieren.

¹ Beispielsweise um zwei HTML-Dateien dasselbe Layout zu geben. Oder um an Weihnachten seiner Webseite ein anderes Layout zu geben, ohne alle HTMLs anpassen zu müssen. 🎅 🎄

SVG (Scalable Vector Graphics)
SVG ist eines der verschiedenen Grafikformate im Web. Das Besondere daran ist, dass es sich um ein Vektorformat (statt einem Rasterformat) handelt, das mit CSS und JavaScript darüber hinaus auch komplett skriptbar ist. Somit lassen sich über HTML hinaus zum Beispiel interaktive Schaubilder und andere grafische Elemente erstellen.
JavaScript
Die Schaltzentrale, die allen Webanwendungen ihre clientseitige Logik gibt, ist JavaScript. Einzelheiten dazu wirst du später noch erfahren. Hier reicht es erst mal zu wissen, dass es sich um eine vollwertige Programmiersprache handelt, die eingebettet in einer Webseite im Browser läuft und alle Inhalte der Seite ändern kann. Zusammen mit ihrer umfangreichen Programmierschnittstelle (API) zur App-Entwicklung lassen sich damit fast beliebige Anwendungen entwickeln.

HTML ist eine sogenannte Markup Language (Auszeichnungssprache). Damit ist gemeint, dass HTML-Dateien einfache Textdateien sind, wie sie mit einem einfachen Texteditor (zum Beispiel Notepad) erstellt und bearbeitet werden können. Neben Buchstaben, Zahlen und Zeilenumbrüchen beinhalten sie noch für Mensch und Computer lesbare Steueranweisungen, die einzelnen Abschnitten eine Bedeutung zuweisen und dem Text dadurch eine Struktur geben. Weitere Beispiele für solche Sprachen sind:

  • TeX und LaTeX
  • ReStructuredText
  • Markdown
  • Textile
  • Wikitext
  • PostScript
  • Rich Text Format
  • …

Auszeichnungssprachen stehen im Gegensatz zu Binärdateien, wie sie von Microsoft Word, Libre Office und anderen Programmen erstellt werden. Denn diese lassen sich nicht in einem Texteditor öffnen und beinhalten auch keinen lesbaren „Quellcode”, der von Hand editiert werden kann. Die Syntaxregeln von HTML sind eigentlich ganz einfach. Es gibt eigentlich nur Tags (englisch „Tägs” ausgesprochen, machmal auch HTML-Elemente genannt), die in spitzen Klammern stehen, und Fließtext. Allerdings muss zwischen drei Arten von Tags unterschieden werden:

  • Öffnende Tags: Bestehen aus einem Befehl in spitzen Klammern und markieren den Anfang eines Bereichs. Zum Beispiel <title>, <body> oder <h1>.
  • Schließende Tags: Zu jedem öffnenden Tag gehört ein schließendes Tag mit genau demselben Namen. Vor dem Namen steht jedoch ein Schrägstrich. Sie markieren damit das Ende eines Bereichs. Zum Beispiel </body>. Bei der Reihenfolge ist ist auf eine korrekte Schachtelung zu achten. Dass heißt die Tags müssen immer in der umgekehrten Reihenfolge geschlossen werden, wie sie geöffnet wurden.
  • Sich selbst schließende Tags: Ein Sonderfall sind die sich selbst schließenden Tags. Dabei handelt es sich um spezielle Tags, auf die kein schließendes Tag folgt. Oft erkennt man sie einem Schrägstrich hinter dem Namen, der in HTML allerdings auch wegbleiben darf. Deshalb sind sowohl <br/> als auch <br> gültige, sich selbst schließende Tags.

Neben dem Namen eines Elements können (außer bei schließenden Tags) innerhalb der spitzen Klammer auch Attribute stehen, die sozusagen Parameter für den jeweiligen Befehl darstellen. Beispielsweise dient das Attribut href beim <a>-Element dazu, das Ziel eines Links zu definieren.

Die Groß- und Kleinschreibung der Tagnamen und Attribute ist egal. Inzwischen hat sich aber die Kleinschreibung als allgemeine Form eingebürgert, obwohl bis Anfang der 2000er-Jahre weitgehend Großschreibung vorherrschte. Things have changed.

Darüber hinaus hat jedes HTML-Dokument einen festen Aufbau, der in der folgenden Abbildung dargestellt ist:

Die erste Zeile sollte immer <!DOCTYPE html> lauten. Dadurch wird definiert, dass es sich um eine HTML5-Datei unter Ausnutzung aller aktuellen Browserfeatures handelt. Früher wurde diese Zeile dazu genutzt, verschiedene, teilweise inkompatible HTML-Versionen zu kennzeichnen. Heutzutage ist dies glücklicherweise nicht mehr erforderlich.

Das erste richtige Tag einer HTML-Datei ist immer das <html>-Tag. Man nennt es auch das Wurzelelement, da sich durch die dadurch immer eine Baumstruktur mit nur einem Oberknoten ergibt. Folgerichtig muss </html> immer das letzte Tag einer HTML-Datei sein. <head>…</head> und <body>…</body> sind dann in genau dieser Reihenfolge die einzigen beiden Elemente auf zweiter Ebene, da sie die (meist nicht sichtbaren) Metadaten sowie den anzuzeigenden Inhalt einer Seite kennzeichnen.

Mehr Regeln gibt es im Prinzip nicht, außer dass die Zeichen < und > nicht außerhalb von Tags vorkommen dürfen. Sie müssen stattdessen durch eine „Entitäten” genannte Ersatzschreibweise ersetzt werden:

Zeichen Ersatzschreibweise
< &lt;
> &gt;
& &amp;

Aufgabe 1: Deine erste HTML-Datei

Führe die folgenden Schritte zur Erstellung einer einfachen Webseite aus:

  1. Kopiere den Quellcode der vorangegangenen Folie in einen Texteditor deiner Wahl.
  2. Speichere die Datei mit der Endung .html ab und öffne sie im Browser.
  3. Ändere ein paar Inhalte in der Datei und schaue dir das Ergebnis an.
  4. Lege eine zweite HTML-Datei an und baue jeweils einen Link von der einen auf die andere ein.
Achte auf den richtigen Zeichensatz der Datei. Das Tag <meta charset="utf-8" /> weist den Browser an, die Datei als UTF-8-Datei zu lesen. Sonst werden die Umlaute und Sonderzeichen nicht richtig dargestellt.

Elemente innerhalb des Kopfbereichs

<title>Meine erste Webseite</title>
Legt den Dokumenttitel fest, der in der Titelzeile des Browsers angezeigt wird.
<meta name="name" content="wert" />
Erlaubt es, verschiedene Metadaten wie den Autor der Seite oder Schlagwörter zu ihrem Inhalt zu definieren. Diese Informationen werden von den Suchmaschinen praktisch aber nicht mehr ausgewertet.
<meta charset="utf-8" />
Sonderform des <meta>-Tags zur Definition des Zeichensatzes der Seite. Der Standardwert lautet utf-8, sollte aber trotzdem immer mitgegeben werden, damit Umlaute und Sonderzeichen richtig dargestellt werden.
<link rel="shortcut icon" href="favicon.svg" />
Ordnet der Seite ein Icon zu, das neben ihrem Titel angezeigt wird.

Strukturgebende Elemente

  • Die meisten HTML-Elemente besitzen kein spezielles Aussehen, sondern erhalten dieses erst durch ein Stylesheet.¹
  • Jedoch unterscheiden sich die Elemente in der Art, wie sie auf dem Bildschirm angeordnet werden:
  • Blockelemente umschließen einen größeren Bereich mit beliebig viel Inhalt.
  • Inline-Elemente sind Teil des Fließtexts und werden innerhalb dessen platziert.

    • ¹ Siehe Folien über CSS weiter hinten

    Anordnung von Blockelementen

    Die meisten HTML-Elemente sind Blockelemente. Sie stehen in der Regel untereinander und nehmen immer eine maximale Breite in Anspruch. Die Breite wird dabei durch das sie umgebende Elternelement begrenzt. Beispiele hierfür sind:

    <div>…</div> <header>…</header> <footer>…</footer> <main>…</main> <section>…</section> <nav>…</nav> <h1>…</h1> <h2>…</h2> <h3>…</h3> <h4>…</h4> <h5>…</h5> <h6>…</h6> <p>…</p>

    Werden Blockelemente ineinander geschachtelt, wird auch ihr Inhalt geschachtelt dargestellt. Blockelemente eignen sich daher hervorragend, die Gesamtstruktur einer Seite zu definieren.

    Anordnung von Inline-Elementen

    Inline-Elemente werden als Teil des Fließtextes angeordnet, ganz so als ob es Buchstaben wären. Sie stehen daher mit dem sie umgebenden Text in einer Linie und werden (wie der Text auch) bei Bedarf in eine neue Zeile umgebrochen. Der Zeilenumbruch kann dabei sogar innerhalb eines Inline-Elements erfolgen, so dass Teile davon am Ende der einen Zeile und der Rest am Anfang der nächsten Zeile stehen. Davon abgesehen richten sich Höhe und Breite nach dem Inhalt, so dass ein Inline-Element immer nur gerade so groß ist, dass es seinen Inhalt umschließen kann. Beispiele für Inline-Elemente sind:

    <span>…</span> <a>…</a> <b>…</b> <u>…</u> <i>…</i> <button>…</button> <input>…</input> <img/>

    Strukturierung des HTML-Codes

    Die größte Kunst beim Erstellen einer HTML-Seite ist es, eine gute Struktur für den HTML-Code zu finden, die einerseits möglichst einfach das gewünschte Layout erzielt und andererseits auch übersichtlich und leicht zu verstehen ist. Damit dies gelingt ist es daher wichtig, die Layoutalgorithmen des Browsers zu verstehen und zu wissen, wie welches Element auf dem Bildschirm angeordnet wird. Zwar kann die Anordnung im Stylesheet nachträglich verändert werden, der gesamte Quellcode wird dadurch aber sehr komplex und fehleranfällig.

    Zusammenspiel mit dem Stylesheet

    Idealerweise sollte der HTML-Code nur die darzustellenden Inhalte und die grobe Struktur der Seite beinhalten. Die Layoutregeln des Browsers sollten dabei so ausgenutzt werden, dass auch ohne Stylesheet die Anzeige näherungsweise schon dem gewünschten Ergebnis entspricht. In dem (oder den) Stylesheet(s) sollten dann kosmetische Anpassungen vorgenommen werden, um die Details des Layouts zu optimieren.

    Allerdings ist dies nicht immer möglich. Immer dann, wenn der voreingestellte Layoutalgorithmus des Browsers nicht das gewünschte Ergebnis liefern kann (beispielsweise wenn mehrere Blockelemente nebeneinander stehen sollen), muss im Stylesheet aktiv ein anderer Layoutalgorithmus ausgewählt werden.     • Tabellarische Darstellung
    • „Sticky” Elemente (bleiben beim Scrollen oben oder unten kleben)
    • „Fixe” Elemente (verändern ihre Position auch beim Scrollen nicht)
    • Flexbox (zur Ausrichtung mehrerer Elemente neben- oder untereinander)

    Block- und Inlineelemente

    Was wir dir bisher noch nicht erklärt haben ist, wie der Browser die HTML-Elemente eigentlich auf dem Bildschirm anordnet. Zeit das nachzuholen. Im Prinzip ist es auch ganz einfach: Da HTML-Seiten ursprünglich ja nur aus formatiertem Text bestanden, geht der Browser zeilenweise von oben nach unten, von links nach rechts vor. Eben genauso, wie wir in der westlichen Welt schreiben und lesen. Dabei unterscheidet der Browser bei jedem Element, ob es sich um ein Blockelement oder um eine Inlineelement handelt. Blockelemente stehen nämlich immer in einer eigenen Zeile und sind immer so breit wie möglich. Vor und nach einem Blockelement fügt der Browser daher automatisch einen Zeilenumbruch ein. Beispiele hierfür sind:

    • Überschriften
    • Absätze
    • Listen und Aufzählungen
    • …

    Inlineelemente hingegen erzeugen keinen automatischen Zeilenumbruch. Sie werden stattdessen auf einer Linie (der Basislinie des Textes) so lange nebeneinander angeordnet, bis rechts kein Platz mehr ist und die folgenden Elemente in die nächste Zeile rutschen. Dies ist auch der Grund, warum Inlineelemente etwas sparsamer mit dem Platz umgehen. Bei ihnen richtet sich nicht nur die Höhe nach ihrem Inhalt (wie bei Blockelementen), sondern auch ihre Breite. Das heißt, Inlineelemente sind immer „so klein wie möglich und so groß wie nötig”. Beispiele hierfür sind:

    • Bilder
    • Links
    • Einzelne Wörter
    • Fettgedruckte Bereiche
    • Kursive Bereiche
    • …

    Allgemeingültige HTML-Elemente

    Natürlich kann man damit jetzt noch nicht so wirklich komplizierte Layouts gestalten. Denn dafür funktioniert das Ganze noch viel zu sehr wie eine alte Schreibmaschine. Eben weil die meisten HTML-Elemente von sich aus kein Aussehen besitzen, sondern einfach nur ihren Inhalt auf dem Bildschirm platzieren. Neben der Anordnung auf dem Bildschirm sind sie aber trotzdem für die Gestaltung einer Seite wichtig, da ihr Aussehen in so genannten Stylesheets sehr flexibel definiert werden kann. Die zwei wichtigsten Vertreter dieser Art sind die Elemente <div> und <span>:

    • <div>…</div>: Definiert ein ganz allgemeines Blockelement. Das Tag hat darüber hinaus keine tiefere Bedeutung. Kritiker behaupten deshalb auch, es stehe für den „diversen Kram”, den die Leute dort hinein packen.
    • <span>…</span>: Definiert ein allgemeines Inlineelement. Genau wie <div> hat es darüber hinaus keine Bedeutung.

    Da der Quellcode einer Seite mit vielen <div>-Elementen schnell unübersichtlich wird, wurden mit HTML5 eine ganze Reihe weiterer Tags eingeführt, die sich genau gleich verhalten. Nur mit dem Unterschied, dass man an ihrem Namen ihre ungefähre Bedeutung ablesen kann:

    • <header>…</header>: Definiert den Kopfbereich der Seite, bzw. einen Bereich der oberhalb eines anderen steht, mit diesem aber fest verbunden ist.
    • <footer>…</footer>: Definiert den Fußbereich der Seite, bzw. einen Bereich der unterhalb eines anderen steht.
    • <section>…</section>: Unterteilt einen Bereich in beliebig viele, eigenständige Abschnitte.
    • <nav>…</nav>: Kennzeichnet einen Navigationsbereich. Also einen Bereich, in dem anklickbare Elemente enthalten sind, mit denen man sich innerhalb der Seite bewegen kann.
    • Und so weiter: Es gibt noch einige mehr.

    Damit lässt sich eine Seite beliebig unterteilen, um zusammen mit CSS ihr Layout zu definieren. 👍

    Wie du ja schon weißt, wurde HTML ursprünglich als Markupsprache für Textdokumente entworfen, die über das Web auf dem Bildschirm aufrufbar sein sollten. Über die genaue Darstellung der Dokumente hat man sich dabei gar nicht so viele Gedanken gemacht. Es genügte, wenn jeder Browser ungefähr ähnliche Regeln anwendete, so dass eine Überschrift eben wie eine Überschrift und ein Absatz wie ein Absatz aussah. Heute haben die Browser solche Regeln immer noch fest eingebaut, das Ergebnis entspricht aber kaum modernen Anforderungen. Auch lassen sich mit den eingebauten Regeln nicht mal ansatzweise moderne Webanwendungen oder sonstige, anspruchsvolle User Interfaces realisieren.

    Hier kommt CSS ins Spiel, weil wir damit nicht nur die eingebauten Regeln des Browsers überschreiben, sondern auch ganz eigene Regeln definieren können. Auf diese Weise können wir dann all die wunderbaren Oberflächen erstellen, die wir von Web, Mobile & Co. heute gewohnt sind. Hierfür ist es allerdings wirklich wichtig, die HTML-Dateien gut zu strukturieren, da wir sonst keinen geeigneten Ansatzpunkt für unsere CSS-Regeln haben. Die HTML-Seite auf dieser Folie ist daher bereits im HTML-Code in die folgenden drei Bereiche eingeteilt:

    1. Kopfbereich
    2. Hauptteil
    3. Fußzeile

    Innerhalb des Kopfbereichs ist nur eine Überschrift. Im Hauptteil sind jedoch mehrere <div>-Container, in denen sich jeweils ein Bild und ein Text befinden. Der Footer hingegen enthält einen Copyrightvermerk und eine eigenartige Liste in einem <nav>.

    Schaust du dir zunächst auf dem Reiter „Aussehen” das Ergebnis ohne Stylesheet an, sieht die Seite noch ziemlich unfertig aus. Doch schon eine kleine Hand voll, gut ausgesuchter CSS-Regeln und derselbe HTML-Code verwandelt sich in die Version im Reiter „Neues Aussehen”. Am HTML sind hierfür, von der Einbindung des Stylesheets abgesehen, keine Anpassungen notwendig. Dabei gibt es sogar kleine, interaktive Spielereien, wie der Hover-Effekt der Bilder und Navigationslinks.

    Im Reiter „Stylesheet einbinden” siehst du, dass wir nur eine kleine Zeile in den HTML-Code einbauen müssen, um das Stylesheet zu laden. Diese befindet sich immer im <head> und lautet:

    <link rel="stylesheet" href="style.css" />

    Die CSS-Datei haben wir hier style.css genannt, du kannst aber natürlich auch jeden anderen Dateinamen verwenden, so lange du Leer- und Sonderzeichen (sofern sie im Dateinamen überhaupt vorkommen) durch die richtige Ersatzschreibweise ersetzt. Auch musst du natürlich auf den Ablageort des Stylesheets achten. So, wie es hier angegeben wurde, muss es im selben Verzeichnis wie die HTML-Datei liegen. Gerne kannst du dabei auch mehrere Stylesheets einbinden, indem du einfach mehrere <style>-Elemente einbaust.

    Es gibt noch zwei andere Arten, wie man CSS-Anweisungen mit HTML kombinieren kann. Diese zeigen wir dir auf der nächsten Folie. Für den Moment solltest du aber erst hier weiter lesen. 🙂

    Die Stylesheet-Syntax

    Auf der Folie siehst auch schön die Syntax eines CSS-Stylesheets. Sie ist nicht wirklich kompliziert, man muss nur ein paar einfache Grundregeln kennen. Zum Beispiel, dass jedes Stylesheet immer die folgenden Dinge enthält:

    • Selektoren: Das ist das, was vor den geschweiften Klammern steht. Damit werden die HTML-Elemente ausgewählt, die man gestalten will.
    • Die geschweiften Klammern: Sie fassen die CSS-Eigenschaften zusammen, die auf die ausgewählten HTML-Elemente angewendet werden sollen. Anders als bei Java darf man die geschweiften Klammern allerdings nie weglassen, auch wenn man nur eine einzelne CSS-Eigenschaft definieren will.
    • Die CSS-Eigenschaften: Innerhalb der geschweiften Klammern stehen dann die Eigenschaften mit dem Aussehen der HTML-Elemente. Die Syntax ist dabei immer Eigenschaft, Doppelpunkt, Wert, Strichpunkt.

    Im Beispiel nicht sichtbar sind Kommentare. Sie werden ähnlich wie in Java mit /* ... */ gekennzeichnet. Die aus Java bekannte Kurzform mit // am Anfang einer Zeile ist allerdings nicht erlaubt. 😳 Welche Selektoren es gibt und was die einzelnen Anweisungen bewirken, siehst du auf den nächsten Folien. Doch zuvor, noch ein kurzer Hinweis zu den verwendeten HTML-Elementen.

    Aber ist das auch das richtige HTML?

    Möglicherweise hast du dir beim Betrachten des HTML-Codes die Frage gestellt, ob das HTML so überhaupt in Ordnung ist. Also ob es zum Beispiel okay ist, eine Liste mit <ul> zu definieren, obwohl die Einträge eigentlich gar nicht wie eine Aufzählung aussehen sollen.

    Und tatsächlich, das HTML ist absolut in Ordnung. Nicht, dass man jetzt einfach irgendwelchen Schweinecode schreiben kann, den man mit einem Stylesheet in Form bringt 🐷, sondern der Code ist aus einem ganz anderen Grund okay: Es handelt sich um semantischen HTML-Code. Das heißt, es wurden wirklich nur so viele HTML-Anweisungen verwendet, wie zur Gestaltung der Seite absolut notwendig sind, kein Element ist zu viel. Und jedes Element drückt durch seinen Namen (oder da, wo es kein passendes HTML-Element gibt, durch das class-Attribut) aus, was es darstellen soll. Kurz gesagt: Auch ohne das Ergebnis zu kennen, sieht man dem Code an, was er bewirken soll.

    Deshalb ist es auch okay, die Navigationselemente der Fußzeile in eine ungeordnete Liste zu packen. Denn um genau das handelt es sich ja: Um eine Aufzählung. Nur dass die Einträge nicht untereinander sondern nebeneinander stehen sollen. Trotzdem ist es eine Aufzählung. Zwar ist es im Grunde genommen egal, welches HTML-Element du verwendest, so lange das Ergebnis hinterher so aussieht, wie es soll. Wenn du nicht aufpasst, kann es allerdings sein, dass deine Webseite nur auf visuellen Bildschirmen gut aussieht.

    Äh, Moment mal! Wie kann denn jemand die Seite nicht auf einem Bildschirm anschauen? 😲 Ja, das geht. Viele Menschen sind auf Screenreader und andere Geräte wie Brailezeilen angewiesen, um mit dem Computer umgehen zu können. Wenn deine Navigation nur aus geschachtelten <div>s besteht, können diese Geräte nichts damit anfangen und lesen die Navigation einfach als Fließtext vor. Handelt es sich aber um eine in einem <nav> eingeschlossene Liste, wissen die Geräte, dass es sich um einen Navigationsbereich handelt, der anders oder nur bei Druck auf einen bestimmten Knopf vorgelesen werden muss. Du siehst: Semantisches Markup ist sehr wichtig.

    HTML und CSS in derselben Datei

    <style> innerhalb des <head> wirkt auf die gesamte Seite.
    Mit dem style-Attribut können einzelne HTML-Elemente gestyled werden.

    <!DOCTYPE html> <html> <head> <meta charset="utf-8"> <title>HTML und CSS in einer Datei</title> <meta name="viewport" content="width=device-width, initial-scale=1"> <style> h1 { color: #CD0D0D; text-shadow: 1px 1px 1px grey; margin: 0; } p { margin-top: 0.5em; } </style> </head> <body> <h1>HTML und CSS endlich vereint!</h1> <p> Was lange währt, wird endlich gut. Nachdem sich HTML und CSS nach ihrer Trennung lange Zeit nicht mehr gesehen haben, haben sie heute endlich wieder zueinander gefunden. </p> <p style="color: darkred; font-weight: bold;"> Mit dem style-Attribut kann man gezielt einzelne Elemente gestalten. Das style-Element wirkt hingegen auf die ganze Seite und sollte daher eher im head stehen. </p> </body> </html>

    Wenn es mal schnell gehen soll und du keine zwei Dateien für HTML und CSS anlegen willst, kannst du beides auch in eine Datei packen. Natürlich ist das dann nicht so schön, wie mit getrennten Dateien und du hast nicht all die Vorteile, die die Trennung von HTML und CSS mit sich bringt. Manchmal braucht man das aber auch gar nicht. HTML und CSS können deshalb auf folgende zwei Arten in derselben Datei untergebracht werden:

    • Durch ein <style>-Element im <head> der Seite
    • Durch das style-Attribut, das man jedem HTML-Element geben kann

    Beide Beispiele siehst du auf der Folie. Dort siehst du auch, dass die Syntax innerhalb von <style>…</style> genau dieselbe wie in einer normalen CSS-Datei ist. Wohingegen sich das style-Attribut immer nur auf ein einzelnes HTML-Element auswirkt und deshalb ohne Selektor und geschweifte Klammern auskommt.

    Mische HTML und CSS nur, wenn du gute Gründe dafür hast. Faulheit sollte nicht dazu gehören. 😉

    Wie der Browser eine Seite sieht

    Wir möchten dich an dieser Stelle nicht langweilen. 😴 Für der Verständnis von CSS und später auch JavaScript ist es aber gut zu wissen, wie der Browser eine Webseite sieht. Also, welche Informationen er im Speicher hält, auf die du mit CSS und JavaScript dann zugreifen kannst. Gemeint ist das so genannte Document Object Model.

    Der Name verrät es schon: Es handelt sich um eine Sammlung von Objekten (im Sinne von Objekten, wie du sie aus Java kennst), die alle Inhalte des aktuellen HTML-Dokuments beinhalten. Und da HTML streng hierarchisch aufgebaut ist (es gibt nur ein Wurzelelement und die Elemente müssen korrekt geschachtelt werden), ist auch die Objektstruktur hierarchisch aufgebaut. Man spricht auch von einer Baumstruktur. Aber das weißt ja bereits aus „Algorithmen und Datenstrukturen”.¹

    Wenn wir uns mit JavaScript beschäftigen, wirst du sehen, dass du die Struktur zur Laufzeit verändern kannst und dass die Änderungen sofort auf dem Bildschirm sichtbar werden. Zum Beispiel kannst du einfach neue Objekte in den Baum einhängen, um weitere Inhalte auf dem Bildschirm anzuzeigen. Oder du kannst einzelne Elemente löschen, um sie verschwinden zu lassen.

    CSS arbeitet auch mit dem Document Object Model. Und zwar durch die Selektoren, mit denen man eigentlich die Struktur des DOM-Baums (so wird das Document Object Model oft genannt) abfragt.

    ¹ Und du hast dich schon gefragt, wofür die Vorlesung gut gewesen sein soll. 🤣

    Einfache Selektoren zur Abfrage des DOM

    Auf der Folie siehst du verschiedene Beispiele für die wichtigsten Arten von CSS-Selektoren. Genauer hingesehen erkennst du, dass es eigentlich nur ein paar wenige Grundformen gibt, die immer wieder miteinander kombiniert werden.

    Die einfachste Abfrage ist die Suche aller Elemente, die einen bestimmten Tag-Namen besitzen. Zum Beispiel alle Überschriften 1 oder alle Absätze. Hierfür schreibt man einfach nur den Namen der gesuchten Tags ohne irgendwas davor oder dahinter. div wählt daher alle <div>-Elemente aus und body das <body>, von dem es ja nur eines geben kann.

    Jedem HTML-Element kann man über das id-Attribut eine eindeutige ID zuordnen. Macht man dabei alles richtig, sollte es die ID wirklich nur einmal im gesamten Dokument geben. Auswählen kann man das Element dann durch den Namen der ID mit einem vorangestellten Schweinegatter 🚪 (oder Hash Tag für die Generation Twitter). Zum Beispiel: #logo oder #copyright.

    Will man mehrere Elemente mit möglicherweise unterschiedlichem Namen auswählen, kann man das class-Attribut hierfür verwenden. Mit ihm kann man einem HTML-Element mehrere durch Leerzeichen getrennte Klassen zuordnen. Die Namen sind frei auswählbar. Innerhalb des Stylesheets muss dem Klassennamen nur ein Punkt vorangestellt werden, um ihn abzufragen. Zum Beispiel: .info oder .critical_warning.

    Anstelle der vorgegebenen id- und class--Attribute kann man auch beliebige andere HTML-Attribute abfragen. Das kommt zwar selten vor, ist aber gerade bei der Gestaltung von Eingabeformularen praktisch, um zum Beispiel nicht eingabebereite Elemente anders zu gestalten, als eingabebereite. Hierfür wird einfach der Name des gesuchten Attributs in eckige Klammern geschrieben. Wahlweise kann man noch den Attributwert abprüfen. Beispiele: *[href="beispiel.de"], input[readonly]

    Verknüpfung mehrerer CSS-Selektoren

    Damit kennst du schon alle Grundformen zur Auswahl verschiedener HTML-Elemente. Am Anfang wirst du die folgenden wohl noch gar nicht brauchen. Trotzdem sollst du hier erfahren, wie man die Suche der HTML-Elemente durch Kombination mehrerer Selektoren verfeinern kann.

    Die erste Möglichkeit ist eine UND-Verknüpfung. Das heißt, man sucht nur bestimmte Elemente, die mehrere Bedingungen gleichzeitig erfüllen. Alles was man dazu tun muss, ist die Selektoren ohne Leerzeichen dazwischen hintereinander zu schreiben. Zum Beispiel p.info für alle <p>, mit class="info".

    Eine weitere Möglichkeit ist die ODER-Verknüpfung. Hier sucht man verschiedene Elemente, die eine von mehreren Bedingungen erfüllen. Sie müssen nicht alle Bedingungen erfüllen, eine würde schon völlig reichen. Hierfür schreibt man einfach die Selektoren als eine Art Liste durch Komma getrennt hintereinander. Zum Beispiel h1, h2, h3.

    Zum Schluss gibt es noch die Möglichkeit, Elemente zu suchen, die an einer bestimmten Stelle im Dokument stehen. Also zu prüfen, welches die übergeordneten Elemente eines Elements sind. Eine transitive Suche erreicht man dabei, indem man einfach zwei oder mehr Selektoren durch Leerzeichen getrennt hintereinander schreibt. Beispielsweise bedeutet article .infobox „Suche alle Elemente mit class="infobox", die sich irgendwo innerhalb eines <article> befinden. Das <article>-Element muss nicht das direkte Elternelement sein, indirekt aber schon.”

    Gerne findet die transitive Suche auch mal zu viele Elemente. Nämlich Elemente, die irgendwo tief geschachtelt innerhalb eines anderen Elements stecken. Um dies zu verhindern, kann man die zusammengehörigen Selektoren durch ein Größerzeichen (spitze Klammer) trennen. Dadurch schränkt man die Suche auf nur die direkten Kinder eines Elements ein. Zum Beispiel: section > h1.

    Wie vermisst man die Welt?

    CSS kennt ziemlich viele Maßeinheiten. Auf der Folie sind deshalb gar nicht alle aufgezählt. Die wichtigsten sind aber alle dabei. Bei vielen davon handelt es sich um absolute Einheiten, bei allen anderen um relative Einheiten. Doch was ist damit gemeint?

    • Absolute Einheiten: Diese sind per Definition immer gleich groß. Ein Zentimeter ist ein Zentimeter ist ein Zentimeter. Beispiele: px, pt, cm, mm, …
    • Relative Einheiten: Beziehen sich immer auf einen bestimmten Bezugswert. Häufig handelt es sich dabei um die Schriftgröße oder um die Breite/Höhe eines anderen Elements. Beispiele: em, rem, %, vh, …

    Für die meisten Einheiten reicht die Kurzbeschreibung aus, um zu verstehen, was sie bedeuten. Manche sind dann aber doch ein wenig komisch, so dass wir sie uns lieber nochmal anschauen sollten. Zum Beispiel, was ist bitte schön ein „virtueller Pixel”? Ist ein Pixel denn nicht einfach ein Pixel auf dem Bildschirm? Leider nein. Früher war das mal so. Doch seitdem hochauflösende Bildschirme Standard geworden sind und es Smartphones mit höchst unterschiedlichen Displays gibt, ist es etwas kompliziert geworden. Auslöser war das iPhone 1, das 2007 auf den Mark kam. Es hatte eine Auflösung von 320x480 Pixeln, auf die man sich als Webdesigner einstellen konnte. Im Laufe der Zeit kamen dann aber immer mehr Geräte mit derselben Größe aber mit ganz anderen Auflösungen auf den Markt, auf denen die iPhone-Webseiten plötzlich viel zu klein aussahen. Deshalb haben sich die Browserhersteller einen Trick überlegt:

    Der Browser lügt, was die Bildschirmauflösung angeht! Ja richtig, der Browser lügt. Und zwar bis sich die Balken biegen. 🏗 Tatsächlich gibt es gar keine Möglichkeit, die wahre Bildschirmauflösung abzufragen. Denn jeder Browser meldet einfach eine – nach Ansicht des Herstellers – zum jeweiligen Gerät passende Auflösung und rechnet die Pixelwerte entsprechend um. Dadurch wollte man die alten iPhone-Webseiten retten und den Webdesignern Arbeit ersparen. Tatsächlich erreicht hat man das Gegenteil. Die Einheit px solltest du daher, wenn überhaupt, nur für die Dicke eines Rahmens oder die größe einer Schrift verwenden. Für alles andere solltest du aber besser andere Einheiten verwenden.

    Weitere interessante Einheiten sind em, ex und rem. Sie beziehen sich alle auf die aktuelle Schriftgröße. Ein em wächst also automatisch, wenn du die Schriftgröße des jeweiligen Elements vergrößerst. Für viele Dinge ist das ziemlich praktisch, da man immer ein gutes Ergebnis erhält. Zum Beispiel für Abstände, da der Betrachter bei einer größeren Schrift in der Regel auch einen größeren Abstand erwartet. em und ex richten sich hierbei nach der Schriftgröße des jeweiligen HTML-Elements. rem hingegen nach der allgemeinen Schriftgröße der Webseite, wie sie dem <html>-Element zugewiesen wurde.

    Was ist eigentlich eine Farbe?

    Über Farben, was sie sind, und woraus sie sich zusammen setzen könnte man ganze Bücher schreiben. 📚 Und klar, natürlich gibt es auch Bücher, die nichts anderes beinhalten. 🤓 So ist sicher keine allzu große Überraschung, dass es mehrere Modelle gibt, wie eine Farbe beschrieben werden kann.

    Für den Computer bestehen Farben oftmals aus den drei Grundfarben Rot, Grün und Blau, weil so gut wie jeder Computerbildschirm damit arbeitet. Es handelt sich dabei um die additiven Grundfarben der Lichtmischung, wie sie auch in der Physik Anwendung finden. Wenn du dich in deiner Freizeit mit Malerei beschäftigst, kennst du stattdessen die subtraktiven Grundfarben der Malerei. Diese sind Rot, Grün und Gelb. Hier haben wir es aber mit einem Computer zu tun. 💻

Wie du auf der Folie siehst, gibt es verschiedene Schreibweisen für die drei Farbanteile: Die hexadezimale Form, die etwas kompakter ist, und die rgb()-Form, die einfacher zu verstehen ist. Welche du davon verwendest ist Geschmackssache. 🍉 In älteren Stylesheets wirst du aber eher die Hex-Schreibweise vorfinden, da es sie schon länger gibt.

    Eine Sonderform nimmt die rgba()-Notation ein. Sie drückt genau dasselbe wie rgb() aus, nur dass noch ein vierter Wert für die Deckkraft hinzukommt. Es handelt sich dabei um eine beliebige Kommazahl zwischen 0…1, wobei 0 komplett unsichtbar und 1 komplett deckend ist.

    Bilder für das Web aufbereiten

    Bilder sind für viele Webseiten unverzichtbar. Egal ob als Hintergrundbild, als lustiges Katzenbild oder als echt seriöse Abbildung: Bilder gehören einfach dazu. Falsch eingesetzt können sie aber auch einer der Hauptgründe für lange Ladezeiten und frustrierte Besucher sein. Denn im Web gilt, dass eine Seite so schnell wie nur irgend möglich laden sollte, damit der Besucher nicht das Interesse verliert und einfach weiter surft. Dabei solltest du auch beachten, dass viele Besucher inzwischen viel mit dem Smartphone unterwegs sind und das Problem da noch viel größer ist. Denn mobile Geräte haben oft nur eine schlechte Internetverbindung und lange Ladezeiten sind der Tod eines jeden Akkus. 😒

    Aus diesem Grund gilt für Grafiken im Web im wahrsten Sinne „weniger ist mehr“. Sowohl mit Blick auf die Anzahl als auch ihre Dateigröße. Welche unnötigen Bilder du auf deiner Seite eliminieren kannst, können wir dir natürlich nicht sagen. 😉 Nachfolgend findest du aber ein paar Tipps, wie du wenigstens die Dateigröße klein halten kannst.

    Wähle das richtige Dateiformat

    Noch bevor du eine Grafik erstellst solltest du dir Gedanken darüber machen was sie darstellen soll und welches Dateiformat dafür in Frage kommt. Denn mit dieser einfachen Entscheidung kannst du bereits die größten Einsparungen erzielen oder eben auch den meisten Platz verschwenden. Zwischen folgenden Formaten kannst du dabei wählen:

    PNG (Portable Network Graphics)

    PNG ist sozusagen das Standardformat für Abbildungen im Web und ein moderner Nachfolger für das alte GIF-Format. Wie bei den meisten Formaten handelt es sich um ein sogenanntes Rastergrafikformat, was bedeutet, dass die Bilder Pixel für Pixel gespeichert werden. Vorteile des Formats sind:

    • Verlustfreie Komprimierung der Bilder¹
    • Unterstützung jeder beliebigen Farbtiefe ohne Farbrasterung²
    • Unterstützung eines Alphakanals für halbdurchsichtige Pixel³

    Jedoch ist das Format nicht für alle Arten von Bildern geeignet. Insbesondere fotoähnliche Bilder mit vielen Farben führen zu sehr großen Dateien. Am besten eignet es sich daher für Screenshots, bei denen es auf die Lesbarkeit der Schrift ankommt sowie für Bilder mit wenigen Farben oder mit halbdurchsichtigem Hintergrund.

    ¹ Durch die Komprimierung gehen keine Bildinformationen verloren.

    ² Es wird keine Farbrasterung benötigt, um die Anzahl der Farben zu reduzieren. Andere Formate arbeiten hier mit einer sogenannten Farbpalette und können nur eine bestimmte Anzahl Farben gleichzeitig darstellen, meist nur 256 Stück.

    ³ Zu jedem Pixel wird zusätzlich noch ein Prozentwert gespeichert, wie durchsichtig oder deckend er dargestellt wird.

    GIF (Graphics Interchange Format)
    Das GIF-Format wurde bereits 1988 noch vor dem Web erfunden. Lange Zeit galt es als das Standardformat für den Austausch von Bildern zwischen Computerbenutzern. Im Vergleich zu PNG hat es aber die Nachteile, dass es nur 256 Farben speichern kann und dass keine halbtransparenten Pixel unterstützt werden. GIF-Grafiken mit durchsichtigem Hintergrund haben daher so gut wie immer unschöne Kanten. Jedoch kann man damit einfache Animationen erstellen, was mit PNG nicht möglich ist. Beispiel für ein animiertes GIF. Bildnachweis: Wikipedia
    JPEG (Joint Photographic Experts Group)

    Dies ist das bekannte Standardformat für komprimierte Fotos, das von jeder beliebigen Digitalkamera erzeugt wird. Im Vergleich zu PNG sind die Dateien sehr viel kleiner, die Komprimierung ist dafür aber verlustbehaftet. Durchsichtige Hintergründe sind ebenfalls nicht möglich. JPEG sollte daher das Standardformat sein, wenn du Fotos oder sehr bunte Screenshots einbinden willst.

    Bei Fotos mit weichen Kanten fallen die Kompressionsverluste oft nicht auf, bei harten Kanten und bei Schriftzeichen dafür umso mehr. Zu sehr komprimierte Bilder beinhalten gut sichtbare „Blöckchen” mit verzerrten Farben und in der Nähe von Buchstaben und Wörtern finden sich häufig rauschende Streupixel.

    SVG (Scalable Vector Graphics)

    SVG ist ein XML-basiertes Format für skalierbare Vektorgrafiken (sagt ja schon der Name 🙃). Anstelle einzelner Pixel werden nur Anweisungen gespeichert, anhand derer sich eine Zeichnung rekonstruieren lässt. SVG-Grafiken sehen daher in jeder Größe und Auflösung gestochen scharf aus. Das Format eignet sich daher gut für Schaubilder und andere technische Zeichnungen, richtig angewendet können aber auch ansprechende Zierelemente damit gestaltet werden. Des weiteren hat es noch folgende Vorteile:

    • SVG-Befehle können über das <svg>-Tag direkt in HTML eingebunden werden.
    • Einzelne Elemente einer SVG-Grafik können mit CSS formatiert werden. Man kann damit z.B. die Farbe oder Art einer Linie nachträglich zur Laufzeit ändern.
    • Zusätzlich können die Elemente einer SVG-Grafik auch mit JavaScript angesprochen und manipuliert werden. Man kann also ohne weiteres auch interaktive Grafiken erstellen, die zum Beispiel auf Maus- oder Touchgesten reagieren.

    So klein wie möglich, so groß wie nötig

    Dies ist eine sehr wichtige Regel für Rastergrafiken im Web. Moderne Kameras erzeugen oft extrem große Bilder mit einer sehr hohen Auflösung. Damit eignen sich die Bilder gut für den Druck oder eine Vollbildanzeige auf großen Bildschirmen. Für Webseiten sind sie aber viel zu groß! Bevor du also ein Foto in deine Seite einbindest, solltest du es unbedingt verkleinern. Für die genaue Größe gibt es keine magischen Werte, die immer funktionieren. Letztlich musst du einfach versuchen, einen Mittelweg zwischen kleiner Dateigröße und guter Darstellung zu finden. Folgende Zahlen können dir dabei als Hilfe dienen:

    • Kleine Bilder: ca. 500 Pixel hoch oder breit, ca. 50 kB
    • Mittlere Bilder: ca. 800 bis 1000 Pixel hoch oder breit, ca. 120 kB
    • Große Bilder: ca. 1000 bis 1500 Pixel hoch oder breit, ca. 250 kB

    Diese Werte sind sicher sehr konservativ geschätzt und du wirst sie nicht immer einhalten können. Und je mehr die Bildschirmauflösungen und Internetbandbreiten im Laufe der Zeit zunehmen werden, desto größer können die Zahlen werden. Dennoch solltest du wirklich um jeden Preis versuchen, jedes unnötige kB einzusparen. Denn es gibt immer einen Anwender, der gerade im tiefsten Urwald (oder im kanadischen Hinterland!) sitzt und über eine instabile und mega-langsame Funkverbindung surft, die ständig zusammen bricht (haben wir in Kanada wirklich erlebt). Deshalb solltest du die Anzahl der Pixel immer klein halten, auch wenn die meisten Bildschirme und Mobilgeräte inzwischen Full HD und mehr darstellen können.

    Ein weiterer Trick aus dieser Rubrik ist die Verwendung unterschiedlicher Bilder für unterschiedliche Endgeräte. Dies kannst du entweder mit CSS Media Queries oder mit dem (noch nicht von allen Browsern unterstützten) <picture>-Tag erreichen. Kleinen Geräten mit in der Regel wenig Bandbreite kannst du damit kleinere Bilder senden, als größeren Geräten mit großen Bildschirmen und meistens üppigem Internetanschluss.

    Optimiere die Exportparameter

    Was uns nun endlich zum Screenshot auf der Folie bringt. Gute Grafikprogramme wie The Gimp erlauben es, das Dateiformat beim Speichern bzw. Exportieren eines Bildes nochmal zu tunen, um kostbare kB zu sparen. Teilweise kann sich die Größe einer Datei dadurch um die Hälfte reduzieren. Der Screenshot zeigt daher eine gute Voreinstellung, die du als Grundlage für eigene JPEG-Bilder verwenden kannst:

    • Show preview in image window: Wenn dieses Häkchen gesetzt ist, siehst du im Exportfenster eine Berechnung der erzielten Dateigröße. Gleichzeitig kannst du das Ergebnis im Hauptfenster von GIMP kontrollieren, da dort eine exakte Vorschau angezeigt wird, wie die komprimierte Grafik später aussehen wird. Achte deshalb besonders auf Kompressionsartefakte und entscheide, ob du mit ihnen leben kannst (ohne geht nicht) oder ob du doch lieber ein paar mehr Bytes spendieren willst.
    • Quality: Das ist der wichtigste Parameter. Je niedriger der Wert, desto kleiner die Datei und desto schlechter die Bildqualität. Je nach Grafik kann man auch mal bis 50% oder tiefer gehen, meistens benötigt man dann aber doch so um die 60% bis 80%, damit es noch gut aussieht.
    • Subsampling: 4:2:2 Hierdurch wird die Farbtiefe reduziert, was meistens wenn überhaupt nur auf den zweiten Blick auffällt. Jedoch kann damit in der Regel rund ein Drittel der Dateigröße eingespart werden.
    • Optimize und Progressive: Drücken die Dateigröße nochmals um ein paar kB. Die Auswirkung ist nicht so groß, aber es kommt ja bekanntlich auf jedes Byte an.
    • Save EXIF-Data, Save thumbnail, Save XMP data, Comment: Das sind alles unnötige Zusatzdaten, die der Browser nicht darstellen kann, die aber Platz belegen. Diese Häkchen solltest du für Webgrafiken immer ausschalten.

    Animationen mit CSS

    transition: width 0.7s, height 0.7s;
    Legt fest, welche Eigenschaften animiert werden sollen und wie lange die Animation jeweils dauern soll. Mehrere Animationen können durch Komma getrennt definiert werden. Abgespielt werden den Animationen, wenn sich die jeweilige Eigenschaft verändert. Hierfür braucht man eigentlich JavaScript, einfache Mausereignisse gehen aber auch ohne.
    @keyframes hin-und-her { 0% { transform: rotate(-4deg); } 100% { transform: rotate(4deg); } }
    Keyframe-Animationen werden mit @keyframe definiert, wobei der Animation ein eindeutiger Namen gegeben werden muss. Die einzelnen Zeitpunkte innerhalb der Animation werden in Prozent von 0% bis 100% angegeben. Wie lange die Animation dauert und wie schnell die Zeitpunkte erreicht sind, hängt von der Geschwindigkeit ab, die man ihr später gibt.
    .wackelpeter { animation: hin-und-her 0.7s alternate infinite; }
    Über die animation-Anweisung und ihre Ableger wird eine benannte Keyframe-Animation an ein Element gebunden. Die Animation fängt sofort an zu laufen und muss nicht zuvor aktiviert werden.

    Mit CSS ist es problemlos möglich, einfache Animationen zu erstellen. Klar, die großen Walt Disney Meisterwerke wirst du damit nicht imitieren können, aber für einfache, bewegliche User Interfaces reicht es allemal. 🎥🎞🎬 Hierfür kennt CSS zwei Arten von Animationen:

    • transition
    • keyframes

    Mit der Anweisung transition lassen sich, wie der Name bereits andeutet, die Übergänge von einem Zustand zum anderen animieren. Beispielsweise kann ein Element, wenn die Maus darüber fährt, eine leichte Drehung vollziehen oder einen etwas helleren Hintergrund annehmen. Gerade im Zusammenhang mit JavaScript ist das eine sehr einfache und wirkungsvolle Art, dem Anwender visuelles Feedback zu geben.

    Für komplexere Animationen, bei denen mehrere Werte gleichzeitig verändert werden sollen und die nicht einfach nur von Wert A nach Wert B überblenden, gibt es keyframe-Animationen. Der Begriff Keyframe stammt dabei aus der alten Zeichentrickschule und meint, dass man mehrere Zeitpunkte markieren kann und dann festlegt, welchen Wert bestimmte Eigenschaften jeweils haben sollen.     Wenn du dich bis hierhin durchgearbeitet hast und auch alle Aufgaben bearbeitet hast, kannst du schon ziemlich viel mit CSS bewegen. 👍 Es gibt aber auch viele Layouts, an denen du dir die Zähne ausbeißen wirst. Zum Beispiel, wenn du ein Element horizontal oder vertikal zentrieren willst ohne seine Größe zu kennen.

    Für diesen und andere Fälle wurden in den letzten Jahren sogenannte CSS-Layoutmodule eingeführt. Keine Angst vor dem Begriff, mit Modulen ist hier nur gemeint, dass es Erweiterungen zum CSS-Standard sind, die mit den verschiedenen Werten der display-Anweisung aktivieren werden. Dadurch wird sozusagen eine andere Logik aktiv, wie die Kinder innerhalb eines Elements angeordnet werden, wodurch sich viele Layouts wesentlich eleganter umsetzen lassen. Und das schöne dabei ist, dass man die Layoutmodule nach Belieben kombinieren kann, da sie ja immer nur auf ein Element wirken.

    So nun aber genug der schönen Worte. Wenn du nach dem Studium etwas Zeit hast und dich im Webdesign verbessern willst, willst du dich definitiv mit den folgenden Themen auseinandersetzen:

    • Das Flexbox-Modell
    • Das Grid-Layoutmodul
    Verwechsle das Grid-Layoutmodul nicht mit CSS-basierten Grids, die du im nächsten Kapitel kennenlernen wirst. Die Grundidee ist zwar ähnlich, es sind aber dennoch unterschiedliche Ansätze mit unterschiedlichen Zielen.

    Do & Don't

    Allgemeine Hinweise

    • Überlege dir eine sinnvolle Verzeichnisstruktur für deine Webseite.
    • Lege zum Beispiel eigene Verzeichnisse für Bilder, Stylesheets und JavaScript-Dateien an.
    • Achte bei Bildern darauf, dass sie nicht mehr aus ca. 300 kB oder besser weniger belegen.
    • Achte bei Bildern auf die Auflösung, so dass sie nicht verpixelt aussehen.
    • Wähle generell das richtige Dateiformat für deine Bilder, je nach Art des Bilds.

    Aufbau der HTML-Dateien

    • Schreibe alle HTML-Elemente klein. <GROSSSCHREIBUNG> war damals.
    • Achte auf eine leserliche und vor allem einheitliche Formatierung deiner HTML-Dateien.
    • Verwende semantische HTML-Elemente, die die Bedeutung ihres Inhaltes wiedergeben.
    • Wenn es eine Alternative zum guten alten <div> gibt, verwende sie.
    • Verwende wirklich nur die HTML-Elemente, die du für die Darstellung benötigst.
    • Strukturiere jede HTML-Seite so, dass ihre Reihenfolge auch ohne Stylesheets logisch ist.

    Gestaltung mit CSS

    • Lagere die CSS-Anweisungen immer in externe CSS-Dateien aus.
    • Versuche das style-Attribut in HTML zu vermeiden.
    • Teile größere Stylesheets in mehrere, logisch zusammenhängende Dateien auf.
    • Teste die Darstellung immer auf möglichst vielen Endgeräten mit unterschiedlichen Formfaktoren.
    • Erstelle immer auch ein Print-Stylesheet, so dass die Seite auch ausgedruckt gut aussieht.
    • Übertreibe es mit den optischen Spielereien nicht. Nutze Effekte und Animationen sparsam.
    • Insbesondere sollten die Formatierung und Animationen die Bedeutung eines Elements unterstreichen.
    • Ein dezentes Layout sieht besser aus als ein verspieltes. Immer!
    • Teste deine Webseite auch auf großen Bildschirmen und großen Auflösungen im Vollbildmodus.
    • Achte zum Beispiel darauf, dass Absätze auf großen Bildschirmen nicht zu breit werden.
    • Mache es nicht wie Wikipedia. Erhöhe stattdessen die Schriftgröße auf großen Bildschirmen.
    • Vermeide Layouts mit fester Breite. Das ist einfach so von vorgestern.
