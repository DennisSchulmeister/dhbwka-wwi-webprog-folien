Was sehen wir hier?

Nach dem Studium des letzten Foliensatzes hast du dich sicher gewundert, wie aus dem bisschen HTML, das du da gesehen hast, jemals ein Webseiten-Layout entstehen soll. Ohne große Umschweife wollen wir dir das an dieser Stelle nun zeigen. Mach dir nichts daraus, wenn du den CSS-Quellcode nocht nicht komplett verstehst. Wenn du mit diesen Folien durch bist, kannst du ihn dir ja nochmal anschauen. 🙂

Doch was sehen wir da in der Folie? 🤔 Zunächst einmal siehst du das Grundgerüst einer einfachen Webseite. Schaue dir den HTML-Quellcode ruhig genauer an. Anhand der HTML-Elemente kannst du erkennen, dass es drei große Bereiche gibt:

  1. Kopfbereich
  2. Hauptteil
  3. Fußzeile

Innerhalb des Kopfbereichs ist nur eine Überschrift. Im Hauptteil sind jedoch mehrere <div>-Container, in denen sich jeweils ein Bild und ein Text befinden. Der Footer hingegen enthält einen Copyrightvermerk und eine eigenartige Liste in einem <nav>.

Alles klar soweit? Kannst du dir ungefähr vorstellen, wie die Seite wohl aussehen soll? Wenn ja, dann klicke auf „Aussehen” und schaue nach, ob du recht hast.

Schon gemacht und nun wieder hier? Nun, okay. Vermutlich ist das doch noch nicht ganz, was du erwartet hattest, oder? 🙃 Irgendwie war es aber auch klar, denn der Browser kann ja keine Gedanken lesen. Er kann nur die Regeln anwenden, die wir dir im letzten Foliensatz gezeigt haben. Und die sind halt doch recht einfach …

Und nun bitte mit Stil

An dieser Stelle kannst du ja mal auf „Neues Aussehen” klicken und schauen, wie die Seite mit etwas CSS aussehen könnte. Gleich viel besser, nicht? 😲 Das ist tatsächlich dieselbe Seite, nur dass sie jetzt eben noch ein Stylesheet besitzt. Hast du bemerkt, wie sich die Bilder und die Links im Fußbereich verhalten? Diese Stylesheets scheinen gar nicht so schlecht zu sein. 😎

Im Reiter „Stylesheet einbinden” siehst du, dass wir nur eine kleine Zeile in das HTML einfügen müssen, um das Stylesheet zu laden. Diese befindet sich im <head> und lautet:

<link rel="stylesheet" href="style.css" />

Das ist ja einfach. Und so wie du es inzwischen gewohnt bist, wird damit eine CSS-Datei angesprochen, die im selben Verzeichnis wie das HTML liegt. Sie könnte aber auch in einem anderen Verzeichnis oder auf einem anderen Server liegen. Wobei letzteres würde man in der Praxis eher nicht machen. Was aber durchaus vorkommt ist, dass ein HTML mehrere Stylesheets hat. Hierfür musst du nur mehrere solcher <link>-Elemente einbauen.

Es gibt noch zwei andere Arten, wie man CSS-Anweisungen mit HTML kombinieren kann. Diese zeigen wir dir auf der nächsten Folie. Für den Moment solltest du hier aber erst weiter lesen. 🙂

Die Stylesheet-Syntax

An dem Beispiel siehst auch schön die Syntax eines CSS-Stylesheets. Sie ist nicht wirklich kompliziert, man muss nur ein paar einfache Grundregeln kennen. Eigentlich unterscheidet man nur drei Dinge:

  • Selektoren: Das ist das, was vor den geschweiften Klammern steht. Damit werden die HTML-Elemente ausgewählt, die man gestalten will.
  • Die geschweiften Klammern: Sie fassen immer mehrere CSS-Anweisungen zusammen, die nach einem Selektor stehen. Deshalb darf man sie nie weglassen, auch wenn man nur eine CSS-Anweisungen zuordnen will.
  • Anweisungen: Innerhalb der geschweiften Klammern stehen dann die eigentlichen CSS-Anweisungen. Der Name einer Anweisung steht vor dem Doppelpunkt, ihr Wert danach. Jede Anweisung endet mit einem Strichpunkt.

Im Beispiel nicht sichtbar sind Kommentare. Sie werden ähnlich wie in Java mit /* ... */ gekennzeichnet. Die aus Java bekannte Kurzform mit // am Anfang einer Zeile ist allerdings nicht erlaubt. 😳

Welche Selektoren es gibt und was die einzelnen Anweisungen bewirken, erklären wir die auf den nächsten Folien. Hier brauchst du dir darum noch keinen Kopf machen. 🤗

Aber ist das auch das richtige HTML?

Bis hierhin mag ja alles noch ganz klar sein. Aber vielleicht hast du dir beim Anschauen auch die Frage gestellt, ob das HTML so überhaupt in Ordnung ist. Also ob es zum Beispiel okay ist, eine Liste mit <ul> zu definieren, obwohl die Einträge eigentlich gar nicht wie eine Aufzählung aussehen sollen.

Und tatsächlich, das HTML ist absolut in Ordnung. Nicht, dass man jetzt einfach irgendwelchen Schweinecode schreiben kann, den man halt mit einem Stylesheet in Form bringt 🐷🤣, sondern der Code ist aus einem ganz anderen Grund okay: Es handelt sich um sogenannten „semantischen HTML-Code”. Das heißt, es wurden wirklich nur so viele HTML-Anweisungen verwendet, wie zur Gestaltung der Seite absolut notwendig sind, kein Element ist zu viel. Und jedes Element drückt durch Namen (oder da, wo es kein passendes HTML-Element gibt, durch das class-Attribut) aus, was es darstellen soll. Kurz gesagt: Auch ohne das Ergebnis zu kennen, sieht man dem Code an, was er bewirken soll.

Deshalb ist es auch okay, die Navigationselemente der Fußzeile in eine ungeordnete Liste zu packen. Denn um genau das handelt es sich ja: Um eine Liste. Nur dass die Einträge eben nicht untereinander sondern nebeneinander stehen und keine Aufzählungszeichen beinhalten sollen. Trotzdem ist es eine Aufzählung von etwas. Du könntest jetzt natürlich einwenden, dass es am Ende vom Tag eigentlich völlig egal ist, ob die Navigation als Liste oder durch irgend ein anderes HTML-Element definiert wird, so lange es hinterher so aussieht, wie es soll. Im Prinzip hast du damit auch recht, allerdings nur so lange, wie jemand die Seite wirklich auf dem Bildschirm anschaut.

Wie bitte?! Wie kann denn jemand die Seite nicht auf dem Bildschirm anschauen? 😲 Ja, das geht. Viele Menschen sind auf Screenreader und andere Geräte wie Brailezeilen angewiesen, um mit dem Computer umgehen zu können. Wenn deine Navigation nur aus geschachtelten <div>s besteht, können diese Geräte nichts damit anfangen und lesen die Navigation als Fließtext zusammen mit dem Rest der Seite vor. Handelt es sich aber um eine in einem <nav> eingeschlossene Liste, wissen die Geräte, dass es sich um eine Navigation mit mehreren Auswahlmöglichkeiten handelt. Dementsprechend können sie die Punkte dann auch vorlesen. Du siehst also, semantisches Markup ist sehr wichtig.

Aufgabe 1: Ein kleines CSS-Quiz

Welcher CSS-Selektor wählt …

… alle Absätze der Seite aus?=

  1. div
  2. p.absatz
  3. p > .absatz
  4. p

… alle Elemente der Klasse „info” aus?

  1. info
  2. #info
  3. .info
  4. info()

… das Element mit der ID „page-title” aus?

  1. page-title
  2. #page-title
  3. .page-title
  4. page #title

… alle Elemente der Klasse „active” aus, die sich innerhalb eines <nav>-Elements befinden?

  1. nav .active
  2. nav > .active
  3. nav.active
  4. nav #active

Mit welcher CSS-Anweisung kann man …

… die Schriftart festlegen?

  1. font-size
  2. font-family
  3. font-weight
  4. text-decoration

… ein Hintergrundbild setzen?

  1. background-color
  2. background-image
  3. background-size
  4. background

… den Innenabstand vom Rahmen eines Elements zu seinem Inhalt definieren?

  1. padding
  2. margin
  3. border

… einen Schatten definieren?

  1. border-radius
  2. text-decoration
  3. text-shadow
  4. box-shadow

Lösung: 4, 3, 2, 1 und 2, 2, 2 und 4, 1, 3 und 4

Hoffentlich hast du nicht schon wieder alles vergessen. 😁 Dieses kleine Quiz soll dir helfen, das zu prüfen. 📑

... Dieser Effekt war eine Zeit lang modern und kommt immer mal wieder zum Einsatz, wenn ein einfarbiger Hintergrund zu langweilig aussieht. EIN GROSSES HINTERGRUNDBILD

Beispiel ansehen

... Du hast sicher auch schon immer mal wieder Webseiten gesehen, bei der sich die ganze Musik auf einer einzigen, langen Seite abspielt. Gerade bei Produktpräsentationen ist dies ganz beliebt. Damit die Seite nicht so statisch wirkt, wird dabei oft ein sogenannter Parallax-Scrolling-Effekt eingesetzt, bei dem der Hintergrund anders scrollt als der Vordergrund. Manche Webdesigner verkünsteln sich hier regelrecht, das grundlegende Layout ist aber einfach zu erstellen.
